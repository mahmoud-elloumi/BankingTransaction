Banking Kafka — Système de Transactions Bancaires en Temps Réel

================================================================================

1. Objectif du Projet

Ce projet implémente un système de traitement de transactions bancaires en temps
réel utilisant Apache Kafka comme middleware de messagerie.

Le système permet de :
  • Produire des transactions bancaires (virements, paiements, retraits, dépôts)
  • Valider automatiquement chaque transaction selon des règles métier
  • Détecter les fraudes en temps réel
  • Visualiser les transactions via un dashboard web interactif

================================================================================

2. Architecture du Système

Architecture basée sur un Producer Kafka, un Topic partitionné (3 partitions),
et deux Consumers (Validation + Détection de fraude).

Flux :
  Producer → Topic bank.transactions.pending → Consumers → Dashboard Web

4. Description des Composants

4.1 BankTransaction.java
  Modèle de données représentant une transaction bancaire.
  Champs principaux :
    • transactionId (UUID)
    • senderId / receiverId
    • amount / currency
    • type (DEPOSIT, WITHDRAWAL, TRANSFER, PAYMENT)
    • status (PENDING, APPROVED, REJECTED, FLAGGED)
    • timestamp (LocalDateTime)
    • description

4.2 Serializer / Deserializer
  Conversion des objets Java en JSON (byte[]) et inversement pour Kafka.
  Utilisation du module JavaTimeModule pour LocalDateTime.

4.3 TransactionProducer
  Envoie les transactions vers le topic Kafka.
  Configuration importante :
    • acks=all
    • enable.idempotence=true
    • compression=snappy
    • retries=3

4.4 TransactionConsumer (Validation)
  Règles métier :
    • amount ≤ 0         → REJECTED
    • amount > 50 000 €  → REJECTED
    • sender manquant    → REJECTED
    • Sinon              → APPROVED

4.5 FraudDetectionConsumer
  Détection en temps réel basée sur 4 règles :
    • amount > 10 000 €              → HIGH risk
    • > 3 transactions en 60 secondes → MEDIUM risk
    • Montant rond ≥ 5000            → LOW risk
    • Destinataire inconnu           → LOW risk

4.6 DashboardServer
  Serveur HTTP exposant les endpoints REST :
    • /api/send
    • /api/transactions
    • /api/stats
    • /api/demo
  Accessible via http://localhost:8080

================================================================================

5. Concepts Kafka Utilisés

  • Topic         : canal de messages
  • Partition     : parallélisme et scalabilité
  • Consumer Group: load balancing
  • Offset        : position unique d'un message
  • Idempotence   : garantie exactly-once
  • Mode KRaft    : Kafka sans ZooKeeper

================================================================================

6. Scénario de Démonstration

Exemple : Transaction de 98 000 € vers ACC-999 → FLAGGED car montant élevé,
destinataire inconnu et montant rond suspect.
